## lvalue и rvalue

Както помните от УП, в C++ имаме два вида стойности - lvalue и rvalue. rvalue допълнително се разделя на два подтипа. Най-общо казано, lvalue са всички именовани променливи и референции, а rvalue не са, т.е. са елементи без място в паметта. В таблицата може да видите разграничението:

```c++
f1(int a); // -> приема lvalue и rvalue 
f2(int& a); // -> приема lvalue
f3(const int& a); // -> приема lvalue и rvalue
```

```c++
// валиден код
int x;
int& f() {
	return x;
}

int main() {
	f() = 7;

	std::cout << x;

	return 0;
}
```

Това е възможно, понеже `x` е глобална променлива, която си има адрес в паметта, така че е `lvalue`, а `f()` връща референция към този `lvalue`

## rvalue референция

Най-общо може да се каже, че `rvalue` се отнася за временни обекти, т.е. такива, които са пред изтриване (към края на жизнения си цикъл). Както може да се досетите, това може да се отрази на някои операции, така че е добре да можем да различаваме между `lvalue` и `rvalue`.

Решението е **rvalue референции**, въведени в C++ 11. Те ни помагат да разграничим дали даден обект е временен. За разлика от нормалните референции се означава с &&.

```c++
int named = 86;
int& lvalueRef = named; // както сме го правили винаги досега
int&& rvalueRef = named; // грешка: rvalue референцията очаква временен обект

// int&& rvalueRef = &named; -> това също няма да работи
```

```c++
int i = 2;
int& ref1 = 2 * i; // резултатът от умножение е rvalue, така че не можем да го приравним на lvalue
const int& ref2 = 2 * i; // константна референция може да бъде приравнена на rvalue
int&& rr2 = i * 42; // тук няма проблем, приравняваме rvalue референция към rvalue
```
Както може да видите, `rvalue` референциите могат да сочат само към временни обекти (т.е. такива, които не са `lvalue`)

Ако все пак искаме `rvalue` референция да сочи към `lvalue` обект, можем да го направим с **std::move**
* **std::move(*<обкет>*)** - конвертира `lvalue` *<обект>* в `xvalue`

В примера по-горе:
```c++
int named = 86;
int&& rvalueRef = named; // както казахме не работи

int&& xvalueRef = std::move(named);
```

## Move конструктор и Move operator=

![](/Week07/resources/move_example.png)

Идеята на move операциите е:
1. Ако `this` вече е създаден, т.е. сочи към някаква памет (т.е. ще използваме `operator=`), трябва да освободим тази памет (стандартно `free()`)
2. "крадем" паметта на `other`, като пренасочваме `this` да сочи към блока памет на `other`, вместо да заделяме наново цял блок динамична памет
3. оствяме `other` в такова състояние, така че изтриването му да не навреди на `this` (все пак е временен обект) - най-лесно това става като просто динамичната памет на `other` сочи към nullptr

```c++
class Student {
	char* name;
	int year;

public:
	// това е move конструктор - подобен е на копиращия конструктор, но приема референция към rvalue
  // понеже ще променяме този обект, то тази референция не е константна
	Student(Student&& other) {
    this->name = other.name;
    other.name = nullptr;

    // примитивните член-данни се копират нормално
    this->year = other.year;
	}
	
	// move oprator= - като сигнатура и реализация много подобен на нормалния, с разликата че приема rvalue референция
	Student& operator=(Student&& other) {
		if (this != &other) {
			free(); // стандартно разписан free(), както при копиращите операции

      this->name = other.name;
      other.name = nullptr;

      this->year = other.year;
		}
		
		return *this;
	}
};
```

Както може да видите, операциите по местене се повтарят, така че можем да ги изнесем в помощна функция **moveFrom()** (може да я кръстите както искате, но стандартно се казва така)

Така, когато имаме разписани *Rule of Three/Голяма четворка + Move конструктор + Move assignment оператор* = **Rule of Five/Голяма шестица**

## Синтезирани move операции
Подобно на копиращите операции, компилатора се опитва да създаде автоматично *move операции*, като следва следните правила:
* компилатора се опитва да създаде собствени *move оператори*, ако всяка член-данна може да бъде "преместена". В противен случай може изобщо да не създаде такива операции
* ако има дефинирани копиращ конструктор и operator= (но не и move такива), компилатора използва тях (а това е нещото, което се опитваме да избегнем)
* ако на *move операторите* зададем *=default* и компилатора не може да ги създаде сам, автоматично се маркират с *=delete*
